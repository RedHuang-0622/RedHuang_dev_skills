# 前置审查专家 (Front-Review)

## 目标

在编写任何代码之前，分析当前代码库状态，明确哪些文件需要修改、为何修改、预期影响范围。

## 上下文获取（按优先级尝试）

1. **dev-goal 模式**：工作目录 `docs/YYYY-MM-DD-{模块或功能名}/` — 输出写入工作目录下的 `front-review.md`
2. **独立模式**：需求来自用户直接描述 — 输出生成独立文件
3. 用户会提供一个功能需求或 bug 描述。

## 执行步骤

1. 使用 `grep` 或 Code Search 定位相关代码
2. 读取定位到的文件内容进行确认
3. 追踪 import/引用关系分析依赖链
4. 生成审查报告文件（命名规范见下文）
5. 询问用户确认后进入下一阶段

## 输出目标

| 模式 | 输出路径 |
|------|---------|
| dev-goal 模式 | 写入工作目录 `docs/YYYY-MM-DD-{模块或功能名}/front-review.md` |
| 独立模式 | 生成 `docs/YYYY-MM-DD-功能变更摘要-front-review.md` |

## 输出格式

```markdown
# 前置审查报告

## 需求摘要
[一句话描述需求]

## 影响文件清单
| 文件路径 | 修改类型 | 具体位置 | 修改原因 |
|---------|---------|---------|---------|
| src/xxx.ts | 新增/修改/删除 | L23-L45 | [原因] |

## 依赖分析
- 上游依赖：[哪些模块依赖这些文件]
- 下游影响：[修改后会影响哪些模块]

## 循环依赖检查
- [ ] 确认无新增循环依赖
- [ ] 如存在潜在循环依赖，已标注替代方案（见"建议方案"）

## 风险预估
- [风险1]：[概率] [严重程度]
- [风险2]：[概率] [严重程度]

## 建议方案
[推荐的实现路径，优先考虑低耦合方案]
```

## 暂停点

询问用户："请确认 front-review.md，确认后我将进入 devplan 阶段"

## 打断机制

- 用户拒绝方案 → 记录拒绝原因，返回修正
- 需求不明确 → 列出待澄清点，停止等待
