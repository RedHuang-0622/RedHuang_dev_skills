# 最终审查专家 (Finish-Review)

## 目标

对所有变更进行最后的全面审查，输出结构化的 review 报告。

## 上下文获取（按优先级尝试）

1. **dev-goal 模式**：工作目录 `docs/YYYY-MM-DD-{模块或功能名}/`
   - 从 `devgoal流程.md` 的 G 阶段获取目标与前置审查
   - 从 `devgoal流程.md` 的 O 阶段获取选定方案
   - 从 `code-changes.md` 获取编码变更
   - 从 `test-report.md` 获取测试结果
2. **独立模式**：需存在 `docs/front-review.md` + `docs/plan.md` + `docs/test.md`（保留兼容）
3. **最小模式**：仅需实际代码变更，从 git diff 反推

## 审查维度（五轴检查）

### 1. 正确性 ✅

- [ ] 功能是否符合 plan.md 预期？
- [ ] 边界情况是否都在 test.md 中覆盖？
- [ ] 错误处理是否完善？
- [ ] 接口契约是否被正确实现？

### 2. 可读性 📖

- [ ] 命名是否符合项目规范？
- [ ] 函数是否过长（>50行需要拆分）？
- [ ] 注释是否与代码一致？
- [ ] 魔法数字是否已提取为常量？

### 3. 架构 🏗️

- [ ] 是否遵循现有设计模式？
- [ ] 有无循环依赖？（`go vet` / `madge` 检查）
- [ ] 新增模块是否有清晰接口？
- [ ] 高内聚低耦合：模块职责是否单一？
- [ ] 设计模式使用是否恰当？

### 4. 安全性 🔒

- [ ] 输入是否经过验证？
- [ ] 有无硬编码密钥/密码？
- [ ] SQL/命令注入风险？
- [ ] 敏感数据是否脱敏？

### 5. 性能 ⚡

- [ ] 有无 N+1 查询？
- [ ] 循环内有无不必要的计算/IO？
- [ ] 内存使用是否合理？
- [ ] 有无不必要的内存分配？

## 终止点

输出最终判断和合并建议。

## 打断机制

- 发现严重问题（安全漏洞、数据丢失风险）→ 立即停止，列出问题，等待处理
- 发现循环依赖 → 停止，要求先重构
- 评分低于 B → 警告，用户决定是否接受
